# Sharetribe Mapping

Phase: 3 - Sharetribe Mapping

Purpose:

- Translate the normalized MyYado product model into Sharetribe-native concepts.
- Preserve the old source trail without coupling MyYado to the old implementation.
- Decide whether each core historical concept is adopted, adapted, deferred, or dropped.
- Keep implementation out of scope for this phase.

Scope:

- Historical source concepts are taken from `docs/product/source-index.md`.
- Target concepts are the current Sharetribe Web Template surfaces in this repo.
- Hosted Sharetribe assets should be preferred for content and configuration when they can express
  the product decision.
- Local files listed below are target locations, not proof that implementation has happened.

Gap categories:

- `adopt`: fits Sharetribe cleanly and should be used as-is or with normal configuration.
- `adapt`: fits Sharetribe with renamed fields, simplified modeling, or template configuration.
- `defer`: valid product idea, but not required for the MVP mapping or needs later backend/process
  work.
- `drop`: should not be carried into MyYado.

## Target Data Model Summary

Listing built-ins:

- `title`: public listing name.
- `description`: main story, hospitality context, and traveler expectations.
- `geolocation` and marketplace location: search and map behavior.
- `price`: nightly rate in the marketplace currency.
- `images`: hero and gallery.
- `availability`: native booking availability.
- `author`: Provider reference.

Listing `publicData` fields for Phase 5 candidate configuration:

- `market`: enum; options `nikko`, `kamakura`, `hakone`, `chiba`; indexed; primary filter.
- `collections`: multi-enum; options `temple-town-stays`, `coastal-culture`, `onsen-retreats`,
  `quiet-arrival`, and `hidden-coast`; required, indexed; secondary filter.
- `curationStatus`: enum; options `approved`, `featured`; indexed; default search visibility and
  ranking signal.
- `stayType`: enum; options `ryokan`, `minshuku`, `machiya`, `villa`; detail-only for MVP.
- `maxTravelers`: long; booking validation and detail-only for MVP.
- `bedrooms`: long; detail and optional filter.
- `beds`: long; detail.
- `bathrooms`: long; detail.
- `amenities`: multi-enum; detail and optional secondary filter.
- `bathing`: multi-enum or enum; detail and optional secondary filter.
- `dining`: multi-enum or enum; detail and optional secondary filter.
- `transferAvailable`: boolean; detail and optional secondary filter.
- `languages`: multi-enum; detail and optional secondary filter.

Listing `privateData` candidates:

- `privateArrivalInstructions`: precise arrival notes after transaction confirmation.
- `providerInternalNotes`: provider/platform notes not shown publicly.
- `applicationNotes`: provider intake notes if the EditListingPage doubles as MVP intake.

User/profile fields for Phase 5 candidate configuration:

- `providerBio`: public text for Providers.
- `providerLanguages`: public multi-enum.
- `hospitalityBackground`: public or protected text depending on copy use.
- `travelerPreferences`: private or protected text.
- `dietaryNotes`: protected or private text.
- `accessibilityNotes`: protected or private text.
- `arrivalNeeds`: protected or private text.

Transaction targets:

- MVP stay booking candidate: listing type `nightly-stay`, transaction process
  `default-booking/release-1`, unit type `night`.
- Optional inquiry candidate: listing type `stay-inquiry`, transaction process
  `default-inquiry/release-1`, unit type `inquiry`.
- Phase 4 must choose whether the primary path is inquiry, booking request, or instant booking
  before UI customization.

Search schema candidates:

- Native location search remains the primary search mode.
- Native price filter remains available.
- Native date availability remains available for booking listings.
- Indexed `publicData` fields: `market`, `collections`, and `curationStatus`.
- UI-only fields must not appear as filters unless backed by search schema.
