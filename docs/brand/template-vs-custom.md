# Template Vs Custom

## Customization Rule

Customize only when it:
- reinforces curation
- increases trust
- improves clarity

Do not customize:
- for novelty
- if the template is sufficient
- if the change makes booking, search, or transaction states less clear

## Surface Mapping

| Surface | Approach | Reason |
| --- | --- | --- |
| Homepage | Custom | Must express positioning, markets, collections, trust, and supply quality |
| Search results | Light customization | Native search is strong; add only curation and trust cues |
| Listing page | Moderate | Preserve native booking clarity while improving trust structure |
| Booking flow | Keep template | Payment and request-to-book confidence matter more than novelty |
| Dashboard | Keep template | Native listing, inbox, and transaction management are sufficient for MVP |

## Sharetribe Boundaries

Use Sharetribe-native behavior for:
- authentication
- listing identity
- search
- availability
- checkout
- payments
- transactions
- inbox
- reviews

Consider custom work only after a documented gap is classified as `adapt` and the simpler
Sharetribe-native path is not enough.

Transaction process rule:
Do not change transaction processes without explicit approval and matching Sharetribe backend
configuration.
